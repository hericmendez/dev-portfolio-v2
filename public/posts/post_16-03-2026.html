Save State: da API externa a um sistema próprio de coleta de dados

Alguns projetos começam como experimentos. Outros começam como necessidades pessoais.

O Save State nasceu um pouco dos dois.

Sempre joguei muitos games, principalmente RPGs e jogos longos, e com o tempo comecei a perceber um problema curioso: minha própria memória já não dava mais conta de acompanhar tudo.

Qual foi mesmo aquele RPG que terminei anos atrás?
Em qual plataforma joguei aquele indie incrível?
Quais jogos ainda estão no meu backlog?

Existem sites que fazem algo parecido, mas eu queria algo mais pessoal — um sistema onde eu pudesse registrar minha própria jornada como jogador.

Foi assim que surgiu o Save State, uma aplicação web pensada para registrar, organizar e revisitar todos os jogos que já joguei ou ainda pretendo jogar.

A ideia é simples: criar uma espécie de memória estruturada da jornada gamer.

Cada jogo registrado pode conter metadados como plataforma, status (jogando, finalizado, backlog), desenvolvedores, publishers, data de lançamento e outras informações relevantes.

Mas, como acontece com quase todo projeto de software, a ideia inicial era apenas o começo.

A primeira arquitetura

A primeira versão do Save State foi construída com um objetivo claro: funcionar o mais rápido possível.

Para obter os metadados dos jogos, o projeto consumia diretamente a IGDB API, uma base de dados bastante popular sobre videogames.

Essa abordagem tinha vantagens claras:

  • implementação rápida
  • grande quantidade de dados disponíveis
  • integração relativamente simples

Na prática, bastava fazer uma requisição para a API e obter informações como nome do jogo, capa, data de lançamento e estúdios envolvidos.

Para um protótipo, isso era mais do que suficiente.

Mas conforme o projeto começou a crescer, algumas limitações começaram a aparecer.

Problemas que começaram a surgir

Com o tempo, alguns aspectos da arquitetura começaram a gerar fricção durante o desenvolvimento.

O primeiro deles foi estrutura de código.

Durante a fase inicial, algumas decisões rápidas acabaram levando a:

  • trechos de código duplicado
  • tipagem inconsistente em TypeScript
  • responsabilidades espalhadas em diferentes partes da aplicação

Nada disso quebrava o sistema, mas tornava a evolução do projeto cada vez mais desconfortável.

Outro ponto era a arquitetura geral do projeto.

Inicialmente, o sistema estava dividido entre:

  • um backend em NestJS
  • um frontend em Next.js

Na prática, essa separação acabou criando redundâncias desnecessárias, já que parte das responsabilidades poderia ser resolvida diretamente no próprio ambiente do Next.

Mas o problema mais frustrante estava na dependência da IGDB API.

Apesar de ser uma excelente base de dados, algumas questões começaram a aparecer:

  • jogos com dados incompletos ou inconsistentes
  • limitações de uso da API
  • uma decisão curiosa de design: muitas consultas que naturalmente seriam GET precisavam ser feitas via POST

Isso não impedia o sistema de funcionar, mas criava uma dependência forte de uma API externa sobre a qual eu não tinha controle.

Foi nesse ponto que ficou claro que o projeto precisava de uma mudança estrutural.

O grande refactor

Em vez de apenas corrigir problemas pontuais, decidi aproveitar o momento para realizar um refactor mais profundo na arquitetura do Save State.

Esse refactor incluiu várias mudanças:

  • simplificação da estrutura geral do projeto
  • reorganização das responsabilidades entre frontend e backend
  • padronização das tipagens em TypeScript
  • remoção de redundâncias arquiteturais

Mas a mudança mais significativa foi repensar completamente a forma como o sistema obtém os dados dos jogos.

Uma nova abordagem para os dados

Em vez de depender diretamente de uma única API externa, o Save State passou a evoluir para um modelo diferente: um sistema próprio de coleta e agregação de dados.

Esse sistema, ainda em desenvolvimento, funciona através de scraping estruturado.

A ideia é relativamente simples:

  1. o sistema recebe um termo de busca
  2. consulta múltiplas fontes públicas
  3. extrai os dados relevantes dessas fontes
  4. avalia a confiabilidade das informações
  5. normaliza e estrutura os dados antes de retorná-los

Com isso, o projeto deixa de ser apenas um consumidor de API e passa a ter uma camada própria de ingestão de dados.

Isso traz algumas vantagens importantes:

  • maior controle sobre os dados coletados
  • possibilidade de combinar múltiplas fontes
  • flexibilidade para evoluir o sistema no futuro

Status atual

O mecanismo de scraping ainda está em desenvolvimento, mas já representa uma mudança significativa na arquitetura do projeto.

O Save State continua sendo, em essência, uma aplicação para registrar jogos.

Mas tecnicamente ele está se tornando algo mais interessante: um sistema capaz de coletar, organizar e consolidar conhecimento sobre jogos a partir de múltiplas fontes.

Mais do que um catálogo de jogos

No fim das contas, o objetivo do Save State continua sendo simples: criar um lugar onde eu — e eventualmente outros jogadores — possamos registrar nossa própria história com os jogos que marcaram nossa jornada.

Mas, como acontece com muitos projetos pessoais, o caminho até lá acabou se tornando tão interessante quanto o próprio destino.

Às vezes você começa tentando resolver um problema pequeno… e acaba aprendendo muito mais do que esperava.
